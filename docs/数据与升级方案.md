# 命盘记录：持久性、个保法风险与无缝上云方案

> 2026-06-12 起草。回答三件事：①记录会不会丢 ②在线回收用户信息的合规风险 ③本地→云端的无缝迁移路线。

## 一、记录持久性（实测结论）

记录存在用户**浏览器的 localStorage**（key `hd_charts_v1`），不是服务器。所以：

- **重启 / 重新部署 Fly 应用 → 记录绝对不受影响**（数据根本不在服务器上）。已实测：刷新、跳页、重新部署后记录都在。
- 写入速度非瓶颈：单条 396 字节，单次写入 0.06–0.27ms，容量约 2 万条。

但 localStorage 有**真实的丢失风险**（已加安全网，但需用户知晓）：

| 风险 | 说明 | 现状/缓解 |
|---|---|---|
| 浏览器清缓存 | 用户主动清「网站数据」 | 导出备份 |
| Safari iOS ITP | 7 天不访问该站，Safari 自动清 localStorage | 已 `requestPersist()`（best-effort）+ 提示备份 |
| 存储压力驱逐 | `storage.persisted()=false` 时浏览器可在空间紧张时清 | 同上 |
| **双域名隔离** | `humandesign.zaiyuxingzhe.com` 与 `guanji-humandesign.fly.dev` 是不同 origin，**各存各的**，互不可见 | ⚠ 待办：统一只对外用一个域名（建议自定义域名，fly.dev 仅备用） |
| 换设备/换浏览器 | 无云同步，新设备空白 | 导出备份 → 新设备导入恢复 |
| 隐私模式 | 配额 0，保存抛错 | 已 try/catch + 显式提示「存储不可用」 |

**已落地安全网（f71c8a8）**：导出备份（下载 JSON）/ 导入恢复（按 id+updatedAt 合并去重）/ 申请持久化存储 / 写入容错 / 防连点重复。记录已带 `v`(schema 版本) + `updatedAt`，是未来无损同步的基础。

**建议尽快做的一件事**：对外只宣传一个域名，避免用户在两个域名各存一半记录。

## 二、在线回收用户信息：个保法（PIPL）风险评估

要收集的字段——**姓名 + 出生年月日时 + 出生地 + 性别**——组合后可识别到特定自然人，属于《个人信息保护法》下的**个人信息**；其中若有**不满 14 周岁未成年人**，其信息属**敏感个人信息**，门槛更高。

「回收每个人填的信息」这个说法本身是**最高风险**的做法——未经明确告知和同意就收集、留存用户个人信息，是直接违规。但合规地做是完全可行的，关键守住下面几条：

1. **告知—同意**：上线前必须有《隐私政策》，收集前取得用户同意；涉及敏感信息、对外提供、跨境的，需**单独同意**（专门、可单独勾选，不能混在一揽子条款里）。
2. **最小必要 + 目的限定**：只收实现功能必需的字段，不为「以后可能有用」囤数据。
3. **跨境传输是当前最大雷区**：Fly 服务器在**新加坡（境外）**。把中国大陆用户的个人信息存到境外，触发 PIPL 第 38 条出境机制——需走 ①网信办安全评估 或 ②个人信息保护认证 或 ③标准合同备案 + 个人信息保护影响评估，对小团队成本很高。**务实解法：一旦要落库存用户个人信息，数据库放境内（阿里云/腾讯云大陆区），不要用 Fly 新加坡存个人信息。** Fly 可继续托管静态站，但「带个人信息的后端」分开放境内。
4. **安全措施**：传输/存储加密、访问控制、留存期限、泄露通报机制。
5. **用户权利**：可查询/更正/删除/撤回同意/导出（携带权）。
6. 量大时还需设个人信息保护负责人、做年度合规审计。

**推荐路线（既满足你的目标、又把合规成本压到最低）**：

- **默认本地优先**：不登录的用户，数据只在本机，**根本不出设备 → 不触发 PIPL**。这也是现在的状态，继续保持。
- **云同步做成显式可选的「账号」功能**：用户主动注册/登录、明确同意后才上传，且后端放境内。
- **产品分析另走匿名化**：想了解「用户都是什么类型/角色分布」这类，收**真正匿名化**（去标识、不可逆、不含姓名/精确出生信息）的聚合统计，与可识别的个人记录分库。真正匿名化的数据不受 PIPL 约束。

一句话：**不要"回收"，要"在用户同意下、为明确目的、最小化地、存在境内地"提供云同步**。

## 三、本地 → 云端「无缝丝滑」升级方案

核心设计原则：**所有读写只走 `HDStore` 这一个模块**（四个页面都不直接碰 localStorage）。这就是「无缝」的接缝——将来把 HDStore 的实现从 localStorage 换成「本地缓存 + 云同步」，四个页面零改动。

### 已就位的地基（本次）
- 每条记录带稳定 `id` + `updatedAt` + `v`(schema 版本) → 支持增量同步与冲突解决（取 updatedAt 较新者）。
- `exportJSON / importJSON(合并去重)` → 既是备份，也是「首次登录把本地记录推上云」的迁移原语。

### 三阶段路线（每阶段都可独立上线，用户无感）

**Phase 0（已完成）— 本地优先 + 安全网**
localStorage + 导出/导入备份 + 持久化申请。无后端，零合规负担。

**Phase 1 — 可选账号 + 云同步（要上线在线版时）**
1. 后端放**境内**（如腾讯云轻量 + Postgres），最小 API：`POST /sync`（上行本地变更）、`GET /sync?since=` （下行增量）。
2. 登录用手机号/微信；登录页放《隐私政策》+ 单独同意勾选。
3. HDStore 增加 `syncEnabled` 状态：
   - 未登录 → 行为完全不变（纯本地）。
   - 首次登录 → 自动把本地全部记录 `importJSON` 式推上云（这就是无缝迁移，老用户登录即同步，不丢一条）。
   - 之后 → 每次增删改本地先写、后台按 `updatedAt`/tombstone(`deleted` 标记) 双向同步；离线照常可用，联网自动补传。
4. 删除走**软删除**（`deleted:true` + updatedAt），保证多端删除能正确传播、也可「回收站」找回。

**Phase 2 — 多端/跨端体验**
登录后换设备自动拉回全部记录；冲突按 updatedAt 合并，必要时给「保留两者」。Service Worker 离线缓存（见审查 P2）让 PWA/Capacitor 也能离线读写。

### 与「并入观己 App（任务 #8）」的衔接
Capacitor 版把 HDStore 的存储后端从 localStorage 换成 App 原生存储（Preferences/SQLite），同样只动 HDStore 内部——页面无感。云同步逻辑两端复用。

---
**落地优先级建议**：① 先统一对外域名（防双域名各存一半）② 当前本地优先 + 备份足够支撑免费测试阶段 ③ 真要上「在线回收/同步」，先定境内后端 + 隐私政策 + 同意流程，再开发 Phase 1。
